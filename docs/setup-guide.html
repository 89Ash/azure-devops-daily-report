What you are building

This tool connects to Azure DevOps, pulls work items from your shared queries and area path, analyzes them, and generates a beautiful interactive HTML dashboard you can open in any browser or email to stakeholders.

  • Area Path Report — team-scoped health, KPIs, and trends
  • Shared Queries sidebar — every saved query with red / amber / green health dots
  • Executive Summary — completion %, priority & severity charts, narrative headline
  • Detailed View — filterable table with work item type, due date, created date, age, and more
  • PM Recommendations — actionable insights (velocity risk, assignee blockers, overdue items)

Prerequisites

  • Python 3.9+ — macOS/Linux: python3 --version · Windows: python --version
  • Git (optional) — needed only if you clone; otherwise download the ZIP from GitHub
  • Azure DevOps access — org URL, project name, and read access to work items (for live mode only)
  • Shared queries — at least one saved query under Shared Queries in your ADO project (for live mode)
1

Get the repository

Choose one of the following ways to copy the project to your machine.

Option A — Git clone (recommended)

git clone https://github.com/89Ash/azure-devops-daily-report.git
cd azure-devops-daily-report

Option B — Download ZIP (no Git required)

  1. Open github.com/89Ash/azure-devops-daily-report
  2. Click Code → Download ZIP
  3. Extract the archive
  4. Open a terminal in the extracted project folder (see platform notes below)

macOS / Linux — open a terminal in the project folder

cd path/to/azure-devops-daily-report

Windows — open a terminal in the project folder

  1. Open the extracted folder in File Explorer
  2. Click the address bar, type cmd, and press Enter — or right-click the folder and choose Open in Terminal
  3. Confirm you are in the project directory: 
    cd path\to\azure-devops-daily-report
Viewing this guide: Open docs/setup-guide.html in Chrome, Edge, or Safari to see the full formatted guide and screenshots. The GitHub file preview shows HTML source only.
2

Create a Python virtual environment

A virtual environment keeps this project's dependencies isolated from your system Python.

macOS / Linux

python3 -m venv .venv
source .venv/bin/activate

Windows (Command Prompt or PowerShell)

python -m venv .venv
.venv\Scripts\activate

If python is not found on Windows, try py -m venv .venv and .venv\Scripts\activate instead.

When the environment is active, your terminal prompt shows (.venv).

3

Install dependencies

pip install -r requirements.txt

Only three packages are required: requests, python-dotenv, and jinja2.

4

Try the demo (no Azure DevOps needed)

Verify the dashboard works on your machine using built-in sample data — no credentials required.

python main_demo.py --open

On Windows, if python is not recognized, use py main_demo.py --open instead.

This writes reports/demo-report-YYYY-MM-DD.html and opens it in your default browser.

5

Create an Azure DevOps Personal Access Token

  1. Go to Azure DevOps → User settings → Personal access tokens
  2. Click + New Token
  3. Name it e.g. Leadership Dashboard Read
  4. Set scope to Work Items → Read (Read is sufficient)
  5. Copy the token immediately — you won't see it again
Security: Never commit your PAT to git. Store it only in .env, which is listed in .gitignore. If a token is ever exposed, revoke it in Azure DevOps and create a new one.
6

Configure environment variables

macOS / Linux

cp .env.example .env

Windows (Command Prompt)

copy .env.example .env

Edit .env with your Azure DevOps values:

VariableExampleRequired
ADO_ORG_URLhttps://dev.azure.com/your-orgYes
ADO_PATyour-token-hereYes
ADO_PROJECTMyProjectYes
ADO_AREA_PATHMyProject\MyTeamOptional
ADO_DEFAULT_QUERYPriority ItemsOptional
AZURE_QUERY1My Priority QueryOptional
AZURE_QUERY2My Secondary QueryOptional
REPORT_OUTPUT_DIRreportsOptional

Use backslashes in area paths as shown in Azure DevOps (e.g. Project\Team Name). On Windows, paths in .env use the same format as in the Azure DevOps UI.

AZURE_QUERY1 and AZURE_QUERY2 set which shared query is selected when the dashboard first opens. They are tried in that order when ADO_DEFAULT_QUERY is not set. ADO_DEFAULT_QUERY overrides both if you set it.

Query names must match exactly. The values for AZURE_QUERY1, AZURE_QUERY2, and ADO_DEFAULT_QUERY must match the query name shown in Azure DevOps under Boards → Queries → Shared Queries — same spelling, spacing, and punctuation (e.g. My Priority Items, not the folder path). Copy the name directly from Azure DevOps into your .env file.
7

Generate your first live report

python main.py --open

On Windows, if python is not recognized, use py main.py --open instead.

The script will:

  1. Connect to your Azure DevOps org and project
  2. Load work items from your area path (if configured)
  3. Iterate every Shared Query and analyze results
  4. Write reports/daily-report-YYYY-MM-DD.html
  5. Open the dashboard in your default browser

Troubleshooting

ProblemSolution
Configuration error: Missing required environment variables Copy .env.example to .env and set ADO_ORG_URL, ADO_PAT, ADO_PROJECT
401 Unauthorized PAT expired or wrong scope — create a new token with Work Items (Read)
No shared queries found Create at least one query under Shared Queries in Azure DevOps Boards
Due Date column is blank Ensure work items have Due Date set in ADO; the tool reads Microsoft.VSTS.Scheduling.DueDate
python or python3 not found Install Python 3.9+ from python.org. On Windows, check Add Python to PATH during install, then try py main_demo.py --open
Area Path tab is empty Set ADO_AREA_PATH in .env to your team's area path
Columns blank or wrong (Due Date, Priority, Severity, etc.) Validate that Azure DevOps field reference names match what the tool requests. Fields are listed in config.py (DEFAULT_FIELDS) and mapped in ado_client.py (_normalize_work_item). Key mappings:
  • Due Date → Microsoft.VSTS.Scheduling.DueDate
  • Priority → Microsoft.VSTS.Common.Priority
  • Severity → Microsoft.VSTS.Common.Severity
  • Created → System.CreatedDate
  • Last update → System.ChangedDate
How to verify in Azure DevOps:
  1. Open your project → Project settings → Process (or Organization settings → Process for inherited templates)
  2. Select a work item type (e.g. Bug) → review fields and note each field's Reference name
  3. Compare those names to DEFAULT_FIELDS in config.py — every field the dashboard needs must appear in that list
  4. If your process uses a different reference name (e.g. a custom due-date field), add it to DEFAULT_FIELDS and update the matching key in ado_client.py
Optional API check (replace org and project): GET https://dev.azure.com/{org}/{project}/_apis/wit/fields?api-version=7.1
Wrong query opens on load, or expected query is not selected AZURE_QUERY1, AZURE_QUERY2, and ADO_DEFAULT_QUERY must exactly match a shared query name in Azure DevOps (not the folder path).
  1. In Azure DevOps, go to Boards → Queries → Shared Queries
  2. Select your query and copy its name exactly as shown (watch for extra spaces, & vs and, hyphens, etc.)
  3. Paste that name into AZURE_QUERY1 or ADO_DEFAULT_QUERY in .env
  4. Re-run python main.py --open — the matching query should appear selected in the sidebar
If no name matches, the dashboard falls back to AZURE_QUERY2, then to the first shared query in the list.
